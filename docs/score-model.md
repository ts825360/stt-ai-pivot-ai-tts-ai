# CoolPath AI 점수구조 기획

## 1. 목적

CoolPath AI의 점수는 단순히 가장 빠른 경로를 고르는 점수가 아니다.
목표는 하루 동안 여행자가 얼마나 덜 지치고 쾌적하게 이동할 수 있는지를 계산하는 것이다.

따라서 점수는 다음 질문에 답해야 한다.

> 이 일정은 오늘 하루 동안 사용자가 무리 없이 이동할 수 있는가?

기준 단위는 모두 `분`이다.

## 2. 핵심 관점

CoolPath AI는 이동시간을 하나의 값으로만 보지 않고, 이동을 네 가지 부담 요소로 나누어 본다.

1. 도보시간
2. 대중교통 탑승시간
3. 대기 및 환승시간
4. 최단경로 대비 추가이동시간

도보는 체력 소모가 크고, 대중교통 탑승시간은 회복 가능성이 있다.
대기와 환승은 이동은 아니지만 피로와 불편을 만든다.
추가이동시간은 사용자가 쾌적한 경로를 선택하기 위해 감수해야 하는 시간 손실이다.

## 3. 선행연구 반영 방식

파리 및 일드프랑스 지역 이동 연구에서는 대중교통을 이용한다고 보행이 사라지는 것이 아니라, 대중교통 접근, 환승, 하차 이후 이동 과정에서 보행이 계속 발생한다는 점을 보여준다.

우리는 이 연구를 다음 논리로 반영한다.

- 대중교통은 도보 부담을 완전히 제거하지 않는다.
- 다만 중간에 탑승 시간이 생기기 때문에 도보만 계속하는 것보다 부담이 낮게 환산될 수 있다.
- 대중교통이 포함되면 사용자가 받아들일 수 있는 하루 보행예산을 소폭 보정한다.

기획 기준:

```text
대중교통 포함 이동 시 보행 허용 범위 증가:
1.35km - 0.85km = 0.50km
```

보행 속도는 기존 프로젝트 계산 기준인 `4.7km/h`를 사용한다.

```text
0.50km / 4.7km/h × 60 ≈ 6.4분
```

따라서 대중교통 포함 이동 1회마다 하루 보행예산을 약 `6.5분` 보정한다.

참고 논문:

https://link.springer.com/article/10.1186/s12966-019-0841-2

## 4. 기본 변수

하루 일정 전체를 기준으로 다음 변수를 사용한다.

```text
W  = 하루 총 도보시간
R  = 하루 총 대중교통 탑승시간
Q  = 하루 총 대기/환승시간
D  = 최단경로 대비 추가이동시간
Nt = 대중교통 포함 이동 횟수
```

모든 시간 단위는 `분`이다.

예를 들어 하루 동안 세 번 이동했다면 각 이동 구간의 도보시간, 탑승시간, 대기/환승시간, 추가이동시간을 모두 더해 하루 기준으로 계산한다.

## 5. 하루 보행예산

기본 하루 보행예산은 `40분`으로 둔다.

대중교통이 포함된 이동은 선행연구를 반영해 1회당 `6.5분`씩 보정한다.
다만 대중교통을 많이 탄다고 보행예산이 무한정 증가하면 안 되므로 상한을 둔다.

밸런스 모드 기준 공식:

```text
B = min(70, 40 + Nt × 6.5)
```

의미:

```text
B  = 하루 보행예산
70 = 밸런스 모드에서의 보행예산 상한
40 = 기본 하루 보행예산
Nt = 대중교통 포함 이동 횟수
```

예시:

| 대중교통 포함 이동 횟수 | 하루 보행예산 |
|---:|---:|
| 0회 | 40분 |
| 1회 | 46.5분 |
| 2회 | 53분 |
| 3회 | 59.5분 |
| 4회 | 66분 |
| 5회 이상 | 최대 70분 |

향후 여행 페이스별로 상한을 다르게 둘 수 있다.

| 여행 페이스 | 보행예산 상한 예시 |
|---|---:|
| 여유롭게 | 60분 |
| 밸런스 | 70분 |
| 꽉 채워서 | 85분 |

## 6. 하루 이동부담 환산식

각 이동 방식은 체감 피로도가 다르므로 가중치를 다르게 둔다.

```text
도보시간 × 1.0
대중교통 탑승시간 × 0.5
대기/환승시간 × 0.8
추가이동시간 × 1.3
```

공식:

```text
E = W×1.0 + R×0.5 + Q×0.8 + D×1.3
```

의미:

```text
E = 하루 이동부담 환산시간
```

여기서 `E`는 실제 총 이동시간이 아니라, 피로도를 반영해 환산한 이동부담 시간이다.

## 7. 도보 초과 패널티

도보가 과도하게 많은 일정이 높은 점수를 받지 않도록, 하루 보행예산을 넘은 도보에 추가 패널티를 적용한다.

초과 도보시간:

```text
O = max(0, W - B)
```

의미:

```text
O = 초과 도보시간
```

도보 초과 패널티는 ReLU 방식으로 설계한다.
허용 범위 안에서는 감점하지 않고, 허용 범위를 넘는 순간부터 초과분만 감점한다.

누진 ReLU 패널티:

```text
Pwalk =
0                                if O = 0
O                                if 0 < O ≤ 15
15 + 1.5×(O - 15)                if 15 < O ≤ 30
37.5 + 2.0×(O - 30)              if O > 30
```

이 구조의 목적은 다음과 같다.

- 보행예산 안에서는 감점하지 않는다.
- 조금 초과한 도보는 완만하게 감점한다.
- 도보가 과도하게 많아지면 점수가 빠르게 낮아진다.
- 도보가 압도적으로 많은 일정이 높은 점수를 받는 상황을 막는다.

## 8. 최종 점수 공식

최종 점수는 100점에서 시작해, 하루 이동부담과 도보 초과 패널티를 반영해 계산한다.

```text
Score = clamp(0, 100, round(100 - max(0, E - B) - Pwalk))
```

의미:

```text
100                = 기본점수
max(0, E - B)      = 하루 이동부담이 보행예산을 초과한 만큼의 감점
Pwalk              = 도보가 과도하게 초과된 경우의 추가 감점
clamp(0, 100, ...) = 최종 점수를 0점에서 100점 사이로 제한
round(...)         = 최종 점수를 1점 단위 정수로 표시
```

## 11. 직접선택과 AI자동선택 적용

### 직접선택

사용자가 직접 선택한 장소를 기준으로 후보 경로를 계산한다.
후보 수는 적게 유지하고, 사용자가 고른 장소 안에서 가장 점수가 높은 경로를 추천한다.

```text
직접선택 = 사용자가 선택한 장소 기준 후보 3개 비교
```

### AI자동선택

AI자동선택은 단순히 유명한 장소를 많이 넣는 기능이 아니다.
여행자의 날씨, 이동부담, 보행예산을 기준으로 하루 전체 점수가 높은 조합을 찾는 기능이다.

```text
AI자동선택 = 후보 일정 조합 5개 비교
```

핵심 질문:

> 이 조합을 선택했을 때 오늘 하루 전체 이동피로가 얼마나 남는가?

따라서 AI자동선택은 다음 기준으로 동작해야 한다.

- 너무 멀리 떨어진 장소 조합은 제외한다.
- 도보가 과도하게 많은 일정은 점수를 낮춘다.
- 대중교통을 포함하더라도 대기/환승/추가이동이 큰 경로는 낮게 평가한다.
- 가까운 장소를 묶어 하루 전체 이동부담이 낮은 조합을 우선한다.

## 12. 이 구조의 장점

이 점수구조는 다음 장점을 가진다.

1. 설명 가능하다.
2. 단순 최단거리 앱과 차별화된다.
3. 도보가 많은 일정이 높은 점수를 받는 문제를 방지한다.
4. 대중교통을 타도 보행이 사라지지 않는다는 연구 결과를 반영한다.
5. 직접선택과 AI자동선택의 차이를 명확하게 설명할 수 있다.
6. 여행 페이스에 따라 보행예산 상한을 조정할 수 있다.

## 13. 주의할 점

이 구조는 MVP 기획 기준이다.
운영 서비스에서는 실제 사용자 데이터, 도시별 보행 환경, 날씨, 혼잡도, 교통 API 데이터를 기반으로 가중치와 기준값을 조정해야 한다.

또한 계단, 오르막, 짐 여부는 현재 점수구조에 포함하지 않는다.
이 요소들은 개인차가 크고, 사용자가 오히려 선호할 수도 있기 때문이다.
따라서 현재 MVP에서는 설명 가능성과 유지보수를 위해 제외한다.

## 14. 발표용 요약

> CoolPath의 점수는 단순 이동시간이 아니라 하루 누적 이동부담을 계산합니다. 도보, 대중교통 탑승, 대기/환승, 최단경로 대비 추가이동시간을 각각 다른 피로도로 환산하고, 대중교통이 포함될 경우 보행 허용 범위가 늘어난다는 선행연구를 반영해 하루 보행예산을 보정합니다. 또한 허용 보행예산을 넘는 도보에는 누진 패널티를 적용해, 도보가 과도하게 많은 일정이 높은 점수를 받지 않도록 설계했습니다.
